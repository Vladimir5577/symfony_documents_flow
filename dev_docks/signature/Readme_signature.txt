================================================================================
Электронная подпись документов — итоговая документация
================================================================================

Внутренний ЭДО с двумя уровнями электронной подписи:
  ПЭП  (simple)   — простая ЭП: повторный ввод пароля учётной записи.
  УНЭП (enhanced) — усиленная неквалифицированная ЭП: индивидуальный ключ .p12
                    на флешке пользователя, выпущенный внутренним УЦ проекта.
Архитектура расширяема под третий уровень qualified (УКЭП, КриптоПро) —
он НЕ реализован, см. раздел «ОГРАНИЧЕНИЯ».

План внедрения (контракты, фазы): dev_docks/plans/e_signature_plan.md
SPA API для Next.js:               dev_docks/spa_api/document/Readme_documents_flow.txt (раздел 8)
Мониторинг:                        dev_docks/Readme_monitoring.txt


--------------------------------------------------------------------------------
1. АРХИТЕКТУРА
--------------------------------------------------------------------------------

Крипто: только ext-openssl PHP (RSA-2048, SHA-256, X.509, PKCS#12).
PDF: FPDI + TCPDF (печатная форма, QR через write2DBarcode).
Браузерная подпись: node-forge (importmap, единственная JS-зависимость).
Новых composer-зависимостей нет.

Сервисы (src/Service/Document/Signature/):
  CertificateAuthorityService  — внутренний УЦ: выпуск (.p12 отдаётся ОДИН раз,
                                 приватный ключ существует только в памяти
                                 запроса), отзыв. Пользовательский сертификат:
                                 RSA-2048, срок 365 дней.
  DocumentFreezeService        — заморозка: DOC/DOCX -> PDF (LibreOffice
                                 soffice, headless), сохранение канонического
                                 PDF, SHA-256, генерация verificationCode.
  SigningService               — оркестрация: send-to-signing, signSimple,
                                 signEnhanced, decline, очередь, история,
                                 уведомления, переход в SIGNED.
  SignatureVerificationService — проверка: пересчёт хэша, openssl_verify,
                                 действительность сертификата НА МОМЕНТ
                                 подписания; поиск документа по содержимому.
  SignedFormGenerator          — печатная форма: канонический PDF + лист
                                 подписей (штампы) + QR. Идемпотентна.

Команды:
  app:ca:init            — инициализация корня УЦ (src/Command/CaInitCommand.php)
  app:signature:metrics  — метрики для мониторинга (JSON, см. раздел 8)

Схема данных
------------
user_certificate (src/Entity/Document/UserCertificate.php):
  user, serial_number (unique), subject_dn, certificate_pem, valid_from,
  valid_to, status (active|revoked), revoked_at, revocation_reason,
  issued_by (админ), created_at.
  Приватный ключ пользователя НЕ хранится нигде — ни в БД, ни в файлах.

document_signature (src/Entity/Document/DocumentSignature.php):
  document, signer, level (simple|enhanced),
  document_hash (hex SHA-256 канонического PDF на момент подписания),
  signature_value (base64 RSA-подписи; null для ПЭП),
  certificate (null для ПЭП),
  algorithm ('RSA-SHA256' для УНЭП, 'password-confirmation' для ПЭП),
  signed_at, ip_address, user_agent.
  Unique (document_id, signer_id) — один пользователь подписывает один раз.
  Подписи откреплённые: оригинал не модифицируется, штампы — только на
  генерируемой печатной форме.

document — новые поля:
  signature_level      — уровень, выбранный автором (null = без подписания)
  canonical_file       — имя замороженного PDF в canonical-директории
  canonical_file_hash  — hex SHA-256 канонического PDF
  signed_form_file     — имя файла печатной формы
  verification_code    — 16 hex-символов, для QR и /verify/{code}
  sent_to_signing_at

document_user_recipient — новое:
  role = signer (DocumentRecipientRole::SIGNER)
  signing_order (int, null для других ролей):
    параллельный режим — у всех подписантов 1;
    последовательный — 1, 2, 3, ... Право подписи имеет минимальный
    signing_order среди ещё не подписавших.

Миграция: migrations/Version20260717122013.php (единственная, Фаза 0).

Жизненный цикл документа
------------------------
  создание (автор задаёт signatureLevel + подписантов + порядок)
    -> согласование (существующий флоу) -> APPROVED
    -> «Отправить на подпись» (только автор):
         DOCX->PDF при необходимости, канонический PDF заморожен,
         SHA-256, verificationCode -> статус ON_SIGNING
    -> подписанты по очереди/параллельно (ПЭП: пароль; УНЭП: .p12 в браузере)
    -> все подписали -> SIGNED -> печатная форма (PDF + штампы + QR)
  Отказ ЛЮБОГО подписанта -> REJECTED (обязательная причина, попадает
  в document_history как 'signing_declined: <причина>').
  Замороженные файлы неизменяемы; после ON_SIGNING менять состав
  подписантов и уровень нельзя.

Правила уровня: на документе уровня enhanced ПЭП-подпись запрещена
(document_signing_level_insufficient); УНЭП на документе simple — допустима.


--------------------------------------------------------------------------------
2. КОНТРАКТ БРАУЗЕРНОЙ ПОДПИСИ (УНЭП) — НЕ МЕНЯТЬ
--------------------------------------------------------------------------------

Подписывается HEX-СТРОКА SHA-256-хэша канонического PDF КАК ASCII-БАЙТЫ
(не бинарный хэш и не сам файл). Алгоритм RSA-SHA256 (PKCS#1 v1.5).
Сервер проверяет:
  openssl_verify($hexHashString, $rawSignature, $certPublicKey, OPENSSL_ALGO_SHA256)

Клиент (assets/js/document_sign.js, node-forge):
  1. file input .p12 + пароль контейнера;
  2. forge.pkcs12.pkcs12FromAsn1 — извлечение приватного ключа;
  3. md = forge.md.sha256.create(); md.update(hexHashString, 'utf8');
     privateKey.sign(md) -> base64 -> POST sign/enhanced {certificateId, signature};
  4. ключ и пароль НЕ покидают браузер, ссылки обнуляются после подписания.
Хэш для подписи фронт берёт из GET .../sign/challenge (documentHash) или
data-атрибута страницы подписания.


--------------------------------------------------------------------------------
3. ПРОЦЕДУРЫ УЦ
--------------------------------------------------------------------------------

Инициализация (однократно на окружение)
---------------------------------------
1. Задать env (см. раздел 7): CA_ROOT_CERT_PATH, CA_ROOT_KEY_PATH,
   CA_ROOT_KEY_PASSPHRASE. Директория — ВНЕ webroot (var/ca или /var/ca).
2. docker exec php_container bin/console app:ca:init
   Генерирует корень: RSA-4096, самоподписанный X.509 на 10 лет.
   Существующие файлы НЕ перезаписывает (защита от случайной ротации).
3. Права: директория 0700, ключ 0600, владелец — пользователь php-fpm.
   Passphrase — только в env/секретах, не в git.

Выпуск сертификата пользователю (/admin/certificates, ROLE_ADMIN)
-----------------------------------------------------------------
1. «Выпустить сертификат» -> выбрать пользователя, задать пароль .p12
   (сообщает пользователь, min 8 символов).
2. Сервер генерирует пару RSA-2048, подписывает сертификат корнем (365 дней),
   собирает .p12 и отдаёт файл НА СКАЧИВАНИЕ ОДИН РАЗ. Повторное скачивание
   невозможно — сервер хранит только сертификат (public), приватный ключ
   существует лишь в памяти запроса.
3. Файл записывается пользователем на флешку. Всё аудируется в var/log/ca.log.

Отзыв / перевыпуск
------------------
Отзыв: /admin/certificates -> «Отозвать» + обязательная причина.
  Отозванным сертификатом подписать нельзя (certificate_revoked).
  Подписи, сделанные ДО отзыва, остаются действительными (помечаются
  revokedAfterSigning в деталях проверки).
Перевыпуск: «Перевыпустить» = отзыв старого + выпуск нового одним действием.

Утеря флешки (компрометация ключа)
----------------------------------
1. Пользователь немедленно сообщает администратору.
2. Админ отзывает сертификат с причиной «Утеря носителя» — с этого момента
   ключ бесполезен.
3. Админ перевыпускает сертификат, пользователь записывает новый .p12.
4. Старые подписи остаются валидными (сертификат был действителен на момент
   подписания).

Ротация корневого ключа УЦ
--------------------------
app:ca:init перезапись запрещает — ротация только осознанно:
1. Отозвать все активные пользовательские сертификаты (они выпущены старым
   корнем; проверка старых подписей продолжит работать — сертификат хранится
   в БД целиком).
2. Переместить (не удалять!) старые root_ca.crt/root_ca.key в архив —
   они нужны для исторической верификации цепочки.
3. Задать новый CA_ROOT_KEY_PASSPHRASE, выполнить app:ca:init.
4. Перевыпустить сертификаты всем пользователям через /admin/certificates.


--------------------------------------------------------------------------------
4. ПРОВЕРКА ПОДПИСЕЙ — ТРИ ПОВЕРХНОСТИ
--------------------------------------------------------------------------------

1. Карточка документа (/document/{id}, блок «Подписи»,
   templates/document/partials/_signatures_block.html.twig): список
   подписантов, уровень, дата, № сертификата, итог проверки на лету.
2. /verify — загрузка файла (до 9 МБ, не сохраняется): поиск документа по
   SHA-256 содержимого. Найден -> список подписей + результат проверки;
   не найден/изменён хотя бы на байт -> «документ не найден».
3. /verify/{code} — страница по QR-коду с печатной формы (16 hex-символов).
   Доступ к обеим страницам — любой авторизованный пользователь портала.

Правила проверки (SignatureVerificationService):
  - хэш канонического файла пересчитывается и сверяется с document_hash
    КАЖДОЙ подписи (hash_mismatch — подмена);
  - УНЭП: openssl_verify по сертификату (invalid_signature — чужой ключ);
  - сертификат должен быть действителен НА МОМЕНТ signed_at; отзыв ПОСЛЕ
    подписания подпись не инвалидирует (помечается), отзыв ДО — инвалидирует.

Печатная форма: SPA GET .../signed-form, Twig /verify/{code}/signed-form —
канонический PDF + лист подписей: по штампу на подписанта («Подписано
электронной подписью», уровень, ФИО, дата/время, № сертификата для УНЭП,
хэш) + QR со ссылкой {APP_PUBLIC_URL}/verify/{verificationCode}.
Генерируется при переходе в SIGNED и лениво при запросе, если файла нет.


--------------------------------------------------------------------------------
5. ОГРАНИЧЕНИЯ: ЭТО НЕ УКЭП
--------------------------------------------------------------------------------

Реализованы ПЭП и УНЭП по 63-ФЗ в рамках ВНУТРЕННЕГО ЭДО. Юридическая
сила подписей обеспечивается положением об внутреннем электронном
документообороте организации (соглашение сторон), а не аккредитацией.
НЕ юридически равнозначно собственноручной подписи для внешнего оборота.

Чего нет (и что понадобится для УКЭП):
  - ГОСТ-криптография (ГОСТ Р 34.10-2012) — сейчас RSA;
  - сертифицированное СКЗИ (КриптоПро CSP + плагин/токены);
  - квалифицированные сертификаты аккредитованного УЦ (не внутреннего);
  - метки доверенного времени, CRL/OCSP, форматы CAdES.
Точка расширения заложена: SignatureLevel зарезервирован под case QUALIFIED,
подписи откреплённые — формат хранения не изменится.


--------------------------------------------------------------------------------
6. ИНСТРУКЦИЯ ПОЛЬЗОВАТЕЛЯ
--------------------------------------------------------------------------------

Как подписать ПЭП (паролем)
---------------------------
1. Уведомление «Вам на подпись...» -> открыть документ -> «Подписать»
   (или сразу /document/{id}/sign).
2. Ввести пароль СВОЕЙ учётной записи -> «Подписать».
3. Готово: подпись появляется в блоке «Подписи», следующий по очереди
   получает уведомление.

Как подписать УНЭП (с флешки), по шагам
---------------------------------------
0. Один раз: получить у администратора файл ключа (.p12), записать на
   флешку, запомнить пароль контейнера.
1. Открыть документ -> «Подписать» (уровень документа — УНЭП).
2. Вставить флешку, в поле «Файл ключа» выбрать .p12 с флешки.
3. Ввести пароль контейнера .p12 (не пароль учётки!).
4. «Подписать». Браузер сам: откроет контейнер, подпишет хэш документа,
   отправит подпись на сервер. Ключ с компьютера/флешки никуда не уходит.
5. Ошибки: «не удалось открыть контейнер» — не тот файл или неверный пароль
   контейнера; «сертификат отозван/истёк» — обратиться к администратору
   за перевыпуском.

Как отказаться от подписания
----------------------------
На странице подписания «Отказаться» + обязательная причина. Документ
переходит в REJECTED, автор и подписанты получают уведомления, причина
видна в истории документа.

Как проверить документ
----------------------
- Блок «Подписи» на карточке — статус и валидность каждой подписи.
- /verify — загрузить полученный PDF: система найдёт документ по
  содержимому и покажет подписи. Файл изменён — документ не найдётся.
- Отсканировать QR с печатной формы -> /verify/{code}.

7. ENV-ПЕРЕМЕННЫЕ
--------------------------------------------------------------------------------

CA_ROOT_CERT_PATH        Путь к корневому сертификату УЦ (вне webroot).
                         dev: %kernel.project_dir%/var/ca/root_ca.crt
CA_ROOT_KEY_PATH         Путь к корневому приватному ключу УЦ.
CA_ROOT_KEY_PASSPHRASE   Passphrase корневого ключа. Секрет, не коммитить
                         (в dev стоит заглушка — сменить в prod).
PRIVATE_UPLOAD_DIR_DOCUMENTS_CANONICAL
                         Директория замороженных канонических PDF
                         (/uploads/documents/canonical). Файлы неизменяемы.
PRIVATE_UPLOAD_DIR_DOCUMENTS_SIGNED_FORMS
                         Директория печатных форм
                         (/uploads/documents/signed_forms).
APP_PUBLIC_URL           Публичный URL приложения — база для QR-ссылок
                         {APP_PUBLIC_URL}/verify/{code}.

Также используется LibreOffice (/usr/bin/soffice в php_container) для
конвертации DOC/DOCX -> PDF при заморозке.


--------------------------------------------------------------------------------
8. МОНИТОРИНГ (кратко; полностью — dev_docks/Readme_monitoring.txt)
--------------------------------------------------------------------------------

Стек: Prometheus + Grafana + postgres/nginx-exporter (docker-compose.monitoring.yml).
Съём метрик подписи:
  bin/console app:signature:metrics — JSON: подписания по уровням, отказы,
  документы ON_SIGNING/SIGNED, сертификаты (active/revoked/истекающие 30 дней).
  Подключается внешним агентом (node_exporter textfile + cron).
Логи-события: var/log/signature.log (канал monolog `signature`, JSON) —
  signature.signed / signature.declined / signature.verification_failed
  (reason: hash_mismatch | invalid_signature | ...).
Аудит УЦ: var/log/ca.log (канал `ca`).
Смотреть в первую очередь: certificates_expiring_30d > 0 и любые
verification_failed.

9. МАРШРУТЫ (справочно)
--------------------------------------------------------------------------------

Twig:
  /document/{id}/sign              страница подписания (ПЭП-форма или .p12)
  /document/{id}/send-to-signing   POST, автор, из APPROVED
  /document/{id}/decline-signing   POST, отказ с причиной
  /verify, /verify/{code}          проверка (+ /verify/{code}/signed-form)
  /admin/certificates              УЦ: список, /issue, /{id}/revoke,
                                   /{id}/reissue (ROLE_ADMIN)

SPA (/spa/api/documents-flow, JWT):
  POST /documents/{id}/send-to-signing
  GET  /documents/{id}/signatures
  POST /documents/{id}/sign/simple        {password}
  GET  /documents/{id}/sign/challenge
  POST /documents/{id}/sign/enhanced      {certificateId, signature}
  POST /documents/{id}/decline-signing    {reason}
  POST /verify-file                       multipart, до 9 МБ
  GET  /documents/{id}/signed-form
Коды ошибок и тела — dev_docks/spa_api/document/Readme_documents_flow.txt.

Тесты: tests/Service/Signature/*, tests/Controller/{Admin,Document,SpaApi}/*,
tests/Command/SignatureMetricsCommandTest.php. Запуск: bin/phpunit.
