================================================================================
SPA API — Kanban: история задачи (лента активности)
================================================================================

Контроллер: App\Controller\SpaApi\Kanban\ActivityController
Базовый путь: /spa/api/cards/{cardId}/activities

Авторизация: JWT, ROLE_USER (заголовок Authorization: Bearer <token>).

cardId — числовой идентификатор KanbanCard.

История фиксирует действия над задачей для разбора спорных моментов:
кто и когда переместил задачу, поменял срок/приоритет, назначил исполнителя
и т.п. Каждое значимое действие порождает запись в журнале.

ВАЖНО — журнал пишется СЕРВЕРОМ автоматически.
  Фронтенд НЕ создаёт записи истории. Они появляются как побочный эффект
  обычных запросов (создание/перемещение/изменение карточки, подзадач,
  меток, вложений). У этого ресурса есть только ЧТЕНИЕ (один GET).

Что НЕ логируется (сознательно):
  • Комментарии — сам комментарий уже постоянная запись (виден в чате),
    дублировать его в истории не нужно.
  • Удаление карточки — история привязана к карточке и удаляется вместе
    с ней (каскад), поэтому событие удаления записывать некуда.
  • Вложения с context="chat" — они часть чата, в историю не попадают
    (логируются только вложения context="info"/"description").

Роли (KanbanService::requireRole на доске карточки):

  • Чтение (GET) — минимум KANBAN_VIEWER

  Историю видят все участники доски (тот же уровень, что и просмотр самой
  карточки) — это сделано осознанно: для разбора спора факты должны быть
  видны обеим сторонам, а не только администратору.


--------------------------------------------------------------------------------
ПОЛУЧЕНИЕ ИСТОРИИ (с пагинацией)
--------------------------------------------------------------------------------

  Method : GET
  URL    : /spa/api/cards/{cardId}/activities

  Query-параметры:
    offset — int, смещение для пагинации (по умолчанию 0).

  Сортировка — свежие события сверху (createdAt DESC, при равенстве id DESC).
  Размер страницы фиксированный: 30 записей.
  Права: KANBAN_VIEWER.

curl:

  curl -H "Authorization: Bearer <token>" \
       "http://localhost:8080/spa/api/cards/42/activities"

curl (следующая страница):

  curl -H "Authorization: Bearer <token>" \
       "http://localhost:8080/spa/api/cards/42/activities?offset=30"

--- Ответ (200) ----------------------------------------------------------------

{
  "items": [
    {
      "type": "moved",
      "label": "Перемещена в другой столбец",
      "icon": "bi-arrow-left-right",
      "oldValue": "К выполнению",
      "newValue": "Готово",
      "user": { "id": 1, "name": "Иванов Иван" },
      "createdAt": "2026-06-14T15:30:00+00:00"
    },
    {
      "type": "due_date_changed",
      "label": "Изменён срок",
      "icon": "bi-calendar-event",
      "oldValue": "20.06.2026 18:00",
      "newValue": "не задан",
      "user": { "id": 1, "name": "Иванов Иван" },
      "createdAt": "2026-06-14T15:28:11+00:00"
    },
    {
      "type": "created",
      "label": "Задача создана",
      "icon": "bi-plus-circle",
      "oldValue": null,
      "newValue": null,
      "user": { "id": 1, "name": "Иванов Иван" },
      "createdAt": "2026-06-14T12:00:00+00:00"
    }
  ],
  "hasMore": false,
  "nextOffset": 3
}

--- Поля ответа ----------------------------------------------------------------

  items       — массив событий (см. поля события ниже).
  hasMore     — bool, есть ли ещё записи за текущей страницей.
                Если true — запросить следующую страницу с offset=nextOffset.
  nextOffset  — int, offset для следующего запроса (offset + кол-во в items).

--- Поля события (элемент items) -----------------------------------------------

  type       — string, машинный код события (см. таблицу ниже).
  label      — string, готовый человекочитаемый заголовок на русском
               (можно показывать как есть, локализация уже на бэке).
  icon       — string, имя иконки Bootstrap Icons (напр. "bi-arrow-left-right").
  oldValue   — string|null, значение ДО изменения (см. таблицу — где заполнено).
  newValue   — string|null, значение ПОСЛЕ изменения.
  user       — object|null, автор действия:
                 { "id": int, "name": "Фамилия Имя" }
               null — если автор был удалён (запись истории сохраняется).
               На фронте null показывать как «Система».
  createdAt  — string|null, ISO-8601 (формат "c"), когда произошло событие.

--- Ошибки ---------------------------------------------------------------------

  404  { "error": "card_not_found" }
  403  AccessDeniedException (нет роли KANBAN_VIEWER на доске)

ТИПЫ СОБЫТИЙ (поле type) И ИХ ЗНАЧЕНИЯ
--------------------------------------------------------------------------------

Перечень из App\Enum\Kanban\KanbanCardActivityType. Колонки oldValue/newValue
показывают, какие поля заполнены для каждого типа.

  type                  label (RU)                       icon                     old / new
  --------------------- -------------------------------- ------------------------ -----------------------------
  created               Задача создана                   bi-plus-circle           — / —
  moved                 Перемещена в другой столбец      bi-arrow-left-right      старый столбец / новый столбец
  renamed               Изменено название                bi-pencil                старое / новое название
  description_changed   Изменено описание                bi-card-text             — / —   (только факт*)
  priority_changed      Изменён приоритет                bi-flag                  старый / новый приоритет**
  due_date_changed      Изменён срок                     bi-calendar-event        старый / новый срок***
  color_changed         Изменён цвет                     bi-palette               старый / новый цвет****
  assignee_added        Назначен исполнитель             bi-person-plus           — / имя исполнителя
  assignee_removed      Снят исполнитель                 bi-person-dash           имя исполнителя / —
  label_added           Добавлена метка                  bi-tag                   — / название метки
  label_removed         Удалена метка                    bi-tag                   название метки / —
  attachment_added      Добавлено вложение               bi-paperclip             — / имя файла
  attachment_removed    Удалено вложение                 bi-paperclip             имя файла / —
  subtask_added         Добавлена подзадача              bi-list-check            — / текст подзадачи
  subtask_completed     Подзадача выполнена              bi-check2-square         — / текст подзадачи
  subtask_reopened      Подзадача снова открыта          bi-square                — / текст подзадачи
  subtask_removed       Удалена подзадача                bi-list-check            текст подзадачи / —
  subtask_assigned      Назначен исполнитель подзадачи   bi-person-plus           — / "Имя (подзадача: …)"
  subtask_unassigned    Снят исполнитель подзадачи       bi-person-dash           "Имя (подзадача: …)" / —
  archived              Задача архивирована              bi-archive               — / —
  restored              Задача восстановлена из архива   bi-arrow-counterclockwise — / —

  *    description_changed: текст описания НЕ хранится (описание бывает очень
       длинным). Логируется только факт изменения — сам текст смотрите в карточке.
  **   priority_changed: значения — человекочитаемые метки приоритета
       ("Низкий"/"Средний"/"Высокий"); если приоритет снят — "не задан".
  ***  due_date_changed: формат "д.м.Г Ч:М" (напр. "20.06.2026 18:00");
       если срок снят — "не задан".
  ****  color_changed: код цвета рамки (primary/success/warning/danger/info/dark);
       если цвет снят — "без цвета".

  subtask_assigned / subtask_unassigned: в значение кладётся имя исполнителя
       и название подзадачи в формате "Фамилия Имя (подзадача: <текст>)".
       При смене исполнителя подзадачи (был один → стал другой) пишутся ДВЕ
       записи: subtask_unassigned (прежний) и subtask_assigned (новый).

  Примечание: тип "comment_added" присутствует в enum, но НЕ используется
  (комментарии в историю не пишутся — см. начало документа). Может встретиться
  только в старых записях, созданных до отключения этого события.


--------------------------------------------------------------------------------
РЕКОМЕНДАЦИИ ПО ОТОБРАЖЕНИЮ
--------------------------------------------------------------------------------

  • Заголовок строки — поле label (готов к показу). Иконка — поле icon.

  • Блок «что изменилось» — по наличию oldValue/newValue:
      - заполнены оба      → "<oldValue> → <newValue>"   (старое зачёркнуто)
      - только newValue    → "<newValue>"                (что-то добавили)
      - только oldValue    → "<oldValue>"                (что-то убрали)
      - оба null           → строку не показывать (тип говорит сам за себя:
                             created / archived / restored / description_changed)

  • Автор и время: user.name (или «Система», если user=null) + createdAt.

  • Пагинация: грузить лениво, по запросу пользователя. Первая страница —
    GET без offset; кнопка «Показать ещё» видна, пока hasMore=true, и шлёт
    запрос с offset=nextOffset, дописывая полученные items в конец списка.


--------------------------------------------------------------------------------
ПРИМЕЧАНИЯ
--------------------------------------------------------------------------------

  • Коды ошибок — машинные строки из App\Controller\SpaApi\SpaApiError
    (card_not_found). Человекочитаемый текст подставляет фронтенд.

  • Аналог в legacy-API (сессионная авторизация, не для SPA):
    /api/kanban/cards/{cardId}/activities — App\Controller\Kanban\Api\
    KanbanCardActivityApiController. Формат ответа идентичен.

  • Записи истории создаются автоматически при действиях в обоих API
    (legacy /api/kanban/... и SPA /spa/api/...), поэтому история одной задачи
    единая независимо от того, через какой фронтенд её меняли.
